                                            BASICS
                                        ----------------
by default javascript is single threaded and synchronous which means each line waits for the previous one to get executed.


                                            CALLBACKS
                                        --------------------

CALLBACK --> function which is passed to other function as a parameter which gets executed when the main function calls it
simple example -->

function greet(name, calback) {
    console.log("Hi " + name);
    callback();
}
function bye() {
    console.log("Bye!");
}
greet("Pratik", bye); // Hi Pratik \n Bye

CALLBACK HELL --> where there are too many nested callbacks which makes code difficult to read and debug
                    can be avoided by using promises, async/await

        SYNCHRONOUS CALLBACK :-
runs immediately / no delay
example :
function greet(name, calback) {
    console.log("Hi " + name);
    callback();
}
function bye() {
    console.log("Bye!");
}
greet("Pratik", bye); // Hi Pratik \n Bye

        ASYNCHRONOUS CALLBACK :-
executes later
example :
function fetchData(callback) {
    console.log("first");

    setTimeout( () => {
        console.log("second");
        callback();
    }, 2000);
}
function process(){
    console.log("Processing data");
}
fetchData(process); // first \n second (after 2 sec)


                                            setTimeout()        
                                    ----------------------------

it is used to execute a function after delay
syntax --> setTimeout(function, delay, ...args);

IMPORTANT NOTE :-
1. javascript does not actually wait
2. setTimeout() goes to web/node api
3. js continues next lines
4. after delay -> callbacl goes to callback queue
5. event loop pushes event
6. then it runs

IMPORTANT BEHAVIOUR :-

console.log("First")
setTimeout( () => {
    console.log("timeout");
}0);
console.log("Second"); // first \n second \n timeout

"even though 0ms is given it still does not run immediately because the event stil goes through event loop"
what ever time delay (lets say t ms ) is given it means it will eun after atleast t sec not immediately

use cases :-
api delays, animations, backend scheduling

IMPORTANT CONCEPT :-

1. var behaviour
for(var i = 0; i < 3; i++) {
    setTimeout( () => {
        console.log(i);
    }, 1000);
}
output --> 3 3 3
the ideal output should be 0 1 2
but the ouptput is 3 3 3 because var is golbal scoped. it does not create multiple i, instead it stores reference of i for each iteration for i = 0, 1, 2 callback is created and when i becomes 3 loop ends. now the i stores the reference of 3. after 1 sec when all calback goes to callback queue and executes console.log(i) three times where the value of i is 3 so 3 gets printed three times. we can fix this by let.

2. let behaviour
for(var i = 0; i < 3; i++) {
    setTimeout( () => {
        console.log(i);
    }, 1000);
}
output --> 0, 1, 2
this is because let is block scoped. for each iteration it creates memory for different i.
Iteration 1 → i = 0 (new memory)
Iteration 2 → i = 1 (new memory)
Iteration 3 → i = 2 (new memory)
so each callback points to different i and prints 0, 1, 2


                                            EVENT LOOP   
                                    ------------------------

EVENT LOOP --> mechanism that manages the execution of asynchronous js
handles asynchronous js by moving tasks from microtasks and callbacks queue to call stack when it becomes empty priroritising microtasks over macrotasks

components :-
1. call stack : where code executes
2. web api : handles web apis, dom events, setTimeout()
3. callback queue : stores setTimeout() callbacks
4. microtask : stores promises(high priority)
5. event loop : manager that moves task to stack

overview :-
        ┌──────────────┐
        │  Call Stack  │  ← Executes code
        └──────┬───────┘
               │
        (Empty?)
               │
        ┌──────▼───────┐
        │ Event Loop   │
        └──────┬───────┘
               │
   ┌───────────┴───────────┐
   │                       │
┌──▼──────────┐     ┌──────▼──────────┐
│ Microtask   │     │ Callback Queue  │
│ Queue       │     │ (setTimeout)    │
└─────────────┘     └─────────────────┘

Priority:
Microtask > Callback Queue

execution flow :-

for the code -->

console.log("first");
setTimeout( () => {
    console.log("callback");
}, 0);
promise.resolve().then( () => {
    console.log("promise");
});
console.log("second"); // first second promice callback

promise is printed before callback because promise is microtask which has more priority than callback which is a macrotask
golden rule :- Microtasks are executed before macrotasks when the call stack is empty.

remember :- synchronous js >> microtask >> macrotask
some tricky questions :-
1.Promise.resolve.then( () => {
    console.log("1");
    setTimeout( () =>{
        console.log("2");
    }, 0);  
});
setTimeout( () => {
    console.log("3")
}, 0);
output : 1 3 2
this is because 
Promise.resolve().then(...) → goes to Microtask Queue
setTimeout(() => console.log("3"), 0) → goes to Web API → Callback Queue
event loop starts working
promise gets executed and now setTimeout inside promise goes to callback queue after the outer setTimeout

2. console.log("A");

setTimeout(() => {
  console.log("B");
  Promise.resolve().then(() => console.log("C"));
}, 0);
Promise.resolve().then(() => {
    console.log("D")
    setTimeout(() => {
        console.log("F")
    }, 0)});
console.log("E"); 
output : A E D B C F


                                            PROMISE
                                    ------------------------

promise : object that represents a future result
it has three state : pending, successful, rejected

simple code -->
const promise = new Promise((resolve, reject) => {
    setTimeout(() => {
        resolve("Data revceived")
    }, 2000);
}); // promise goes to resolve state after 2 sec

whatever parameter we give to :
resoleve, resolve pass it to then
reject, reject pass it to catch and skips all then untill it finds catch

handing errors -->
const promise = new Promise ( (resolve, reject) => {
        let success = false;
        if(success) {
            resolve(success);
        } else {
            reject(success);
        }
});
promise.then((res) => {console.log(res)});
promise.catch((res) => {console.log(res)}); // false


then() :-
it is used to handle the result of resolve problem --> when the promise is successful do this
important rule : whenever you return inside then() it goes to next then()


a simple code to understand the logic of promise chaining -->

function getData() {
    return new Promise((resolve, reject) => {
        setTimeout(() => {
            console.log("Data processed");
            resolve("raw data");
        }, 1000);
    });
}
function processData(data) {
    return new Promise((resolve, reject) => {
        setTimeout(() => {
            console.log("Processing data");
            resolve(data.toUpperCase());
        }, 1000);
    });
}
function updateUI(result) {
    return new Promise((resolve, reject) => {
        setTimeout(() => {
            console.log("Updating ui ", result);
            resolve("Done");
        }, 1000);
    });
}
getData().then((data) => processData(data))
    .then((result) => updateUI(result))
    .then((final) => console.log(final))
    .catch((err) => console.log(err));


                                        ASYNC/AWAIT
                                ---------------------------

a cleaner way to write promise code
async - makes a function return a promise
await - makes a async function wait for a promise

use case :
the above code containing promise chain can be made more readable, simple and cleaner like

const data = await getsData();
const result = await processData(data);
const final = await updateUI(result);
console.log(final);
